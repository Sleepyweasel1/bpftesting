# Unify Captured IP state behind a Capture Store seam

## What to build

Extract the dual-map IPv4/IPv6 branching into a single `CaptureStore` module inside `hold-packet`. All callers (`CapturelistServer`, `spawn_idle_monitor`) receive one handle rather than two typed `Arc<RwLock<BpfHashMap>>` handles. The module exposes a map-agnostic interface (`insert(IpAddr, StateEntry)`, `remove(IpAddr)`, `iter() -> Vec<(IpAddr, StateEntry)>`, `get(IpAddr) -> Result<Option<StateEntry>, CaptureStoreError>`) and hides the `STATEV4`/`STATEV6` split internally.

End-to-end: a `gRPC AddRule` call with an IPv4 or IPv6 address flows through `CaptureStore::insert`, hits the correct eBPF map, and a subsequent `ListRules` returns the IP — all without callers knowing which map was used.

## Acceptance criteria

- [ ] `CaptureStore` struct encapsulates both eBPF map handles; no direct map handle fields remain on `CapturelistServer` or `spawn_idle_monitor`.
- [ ] All four gRPC handlers (`AddRule`, `RemoveRule`, `ListRules`, `ReplayRule`) compile and pass existing integration tests after the refactor.
- [ ] `CaptureStore::iter()` is an eager snapshot (`Vec<(IpAddr, StateEntry)>`) and `CaptureStore::get()` supports direct IP lookup.
- [ ] `CaptureStore` has unit tests covering insert/remove/iter for both IPv4 and IPv6 addresses, exercised without a live eBPF environment.
- [ ] The `CaptureStore` seam is documented with a single coherent interface comment; `CONTEXT.md` is updated if any new term is introduced.

## Agent Brief

**Category:** enhancement
**Summary:** Introduce a `CaptureStore` seam that hides IPv4/IPv6 eBPF map split from userspace callers.

**Current behavior:**
Userspace callers directly carry and branch on two map handles (`STATEV4` keyed by `u32` and `STATEV6` keyed by `u128`). `CapturelistServer` handlers and the Idle Monitor each implement duplicated IPv4/IPv6 branching and map-specific lock handling.

**Desired behavior:**
Userspace callers depend on a single `CaptureStore` handle that encapsulates both eBPF maps and routes by `IpAddr` internally. Kernel-side dual maps remain unchanged (this seam is a userspace complexity boundary, not a kernel map unification).

**Key interfaces:**
- `CaptureStore` — a userspace facade owning both state maps behind one unified lock.
- `CaptureStoreError` — typed error surface for map operations instead of stringly-typed errors.
- `CaptureStore::insert(IpAddr, StateEntry)` — inserts by IP family.
- `CaptureStore::remove(IpAddr)` — removes by IP family.
- `CaptureStore::iter() -> Vec<(IpAddr, StateEntry)>` — eager snapshot iteration for both families.
- `CaptureStore::get(IpAddr) -> Result<Option<StateEntry>, CaptureStoreError>` — direct lookup for operator-driven targeted updates.
- `CaptureStore` mode update behavior — for scale-finished events, update mode only (`PassThrough`) and preserve telemetry fields (`last_seen_ns`, `packet_count`). Missing IP is an idempotent no-op.

**Acceptance criteria:**
- [ ] `CaptureStore` exists as a single module in `hold-packet` and is exported via crate API (`lib.rs`).
- [ ] `CapturelistServer` no longer stores direct map handles; it uses one `CaptureStore` handle.
- [ ] Idle Monitor path no longer accepts direct map handles; it uses one `CaptureStore` handle.
- [ ] `CaptureStore` returns typed `CaptureStoreError` values for insert/remove/get/update failures.
- [ ] `CaptureStore::iter()` returns an eager combined snapshot as `Vec<(IpAddr, StateEntry)>`.
- [ ] `CaptureStore::get()` supports direct per-IP lookup for future operator completion flow.
- [ ] Scale-finished mode update path is idempotent when IP is absent and only changes mode when present.
- [ ] Existing gRPC integration tests pass after refactor.
- [ ] New unit tests cover IPv4 and IPv6 insert/remove/iter/get behavior without requiring a live eBPF runtime.

**Out of scope:**
- Unifying kernel eBPF map key types into a single map.
- Introducing trait-based backend abstraction for multiple storage backends.
- Changing map capacities, kernel map layout, or replay/staging packet ID semantics.
