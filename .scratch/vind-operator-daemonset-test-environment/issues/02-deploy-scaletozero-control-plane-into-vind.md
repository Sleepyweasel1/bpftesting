# Deploy ScaleToZero control plane into vind

## What to build

Deploy the ScaleToZero control plane into the repo-owned `vind` environment so later integration slices can exercise the Reconciler and the hold-packet daemon together. This slice should package and apply the Kubernetes resources needed to run the `hold-operator`, register the `ScaleToZero` CRD, and run the hold-packet daemon as a DaemonSet, with a smoke verification path that proves the environment is ready for future tests.

## Design decisions (locked)

| # | Decision | Choice |

| 1 | Entrypoint | Single `mise run vind-stz-up` with internal smoke phase; `mise run vind-stz-down` for teardown |

| 6 | Manifest layout | `hold-packet/deploy/vind/` split files: `namespace.yaml`, `crd-scaletozero.yaml`, `rbac.yaml`, `hold-operator-deployment.yaml`, `hold-packet-daemonset.yaml` |
| 7 | Network interface | Configurable via `HOLD_PACKET_IFACE` env var, default `eth0`; set in DaemonSet manifest and overridable in the task |
| 8 | CRD source | Generated at deploy time by a dedicated Rust binary that calls `ScaleToZero::crd()` and prints YAML; task runs the generator then applies the output |
| 9 | RBAC scope | Strict current-scope: cluster-scoped list/watch on `scaletozero.scale.sleepy.com`; patch/update on `scaletozero/status`; no deployment or service mutations |
| 10 | Smoke failure | Preserve resources; print pod list, events, and describe for operator Deployment and hold-packet DaemonSet |
| 11 | Destroy | `vind-stz-down` deletes in reverse order: DaemonSet → operator Deployment → RBAC → Namespace → CRD |
| 12 | Image tags | Fixed `vind-local` tags; always rebuilt and reloaded on `vind-stz-up` |

## Acceptance criteria

- [ ] `mise run vind-stz-up` builds `hold-operator:vind-local` and `hold-packet:vind-local` images locally, loads them into the vind cluster, then applies resources in this order: Namespace → CRD (generated from `ScaleToZero::crd()`) → RBAC → hold-operator Deployment → hold-packet DaemonSet.
- [ ] `mise run vind-stz-up` is idempotent against a freshly created `vind` environment and against a cluster that already has the resources.
- [ ] Smoke checks pass: `kubectl get crd scaletozero.scale.sleepy.com` succeeds; hold-operator Deployment reaches `Available`; hold-packet DaemonSet desiredNumberScheduled equals numberReady; all within a 120 s timeout.
- [ ] On smoke failure, the task preserves all resources and prints pod list, events, and describe output for both the Deployment and DaemonSet.
- [ ] `mise run vind-stz-down` removes DaemonSet → Deployment → RBAC → Namespace → CRD in that order; operation is a no-op if resources are already absent.
- [ ] All Kubernetes manifests live under `hold-packet/deploy/vind/` as five split files.
- [ ] A Rust CRD-generator binary (in the hold-operator crate or workspace) prints the `ScaleToZero` CRD YAML; the deploy task invokes it directly without external schema tools.
- [ ] hold-packet DaemonSet uses `HOLD_PACKET_IFACE` env var (default `eth0`) for interface selection.
- [ ] The slice does not add behavior-specific scale-to-zero test cases; it only proves the environment is ready for them.

## Agent Brief

**Category:** enhancement
**Summary:** Deploy the ScaleToZero control plane (CRD, RBAC, hold-operator, hold-packet DaemonSet) into the vind dev cluster with a smoke-ready entrypoint.

**Current behavior:**
The `vind-up` / `vind-down` mise tasks create and destroy a bare `hold-packet-dev` vcluster. No control-plane resources are deployed into it — the `hold-operator` and hold-packet daemon have no Kubernetes packaging, and no CRD or RBAC manifests exist.

**Desired behavior:**
`mise run vind-stz-up` builds both container images locally, loads them into the running vcluster, applies all control-plane resources in dependency order, then blocks until smoke checks pass. `mise run vind-stz-down` tears everything down in reverse order including the CRD. The pair is idempotent and self-contained — no registry access is required.

**Key interfaces:**

- `ScaleToZero::crd()` (via `kube::CustomResourceExt`) — the Rust generator binary must call this to emit the CRD YAML; the exact type is defined in the `hold-operator` crate's `stz` module
- `ScaleToZeroSpec` / `ScaleToZeroStatus` — the CRD schema is derived from these types; the generator must import them so the emitted schema stays in sync
- `hold-packet` CLI args: `--iface <string>` (default `eth0`) and `--grpc-addr <addr>` (default `[::]:50051`) — the DaemonSet spec must pass at minimum `--iface` from the `HOLD_PACKET_IFACE` env var
- `hold-operator` binary: reads Kubernetes credentials from the environment via `Client::try_default()`; no flags needed
- vcluster image loading: images must be loaded into the `hold-packet-dev` vcluster using `vcluster cp` or an equivalent mechanism that works with the Docker driver — confirm the correct tool/command by inspecting the vcluster version in mise.toml

**Acceptance criteria:**

- [ ] A Rust binary (separate from the `hold-operator` daemon binary) prints the `ScaleToZero` CRD YAML to stdout; `mise run vind-stz-up` calls it and pipes the output to `kubectl apply`
- [ ] `mise run vind-stz-up` builds `hold-operator:vind-local` and `hold-packet:vind-local` images using `docker build` and loads them into the cluster, then applies resources in this order: Namespace `hold-system` → CRD → RBAC → hold-operator Deployment → hold-packet DaemonSet
- [ ] `mise run vind-stz-up` is idempotent: succeeds on a fresh cluster and on a cluster where the resources already exist
- [ ] Smoke checks pass within 120 s: `kubectl get crd scaletozero.scale.sleepy.com` returns 0; hold-operator Deployment in `hold-system` reaches `Available`; hold-packet DaemonSet `desiredNumberScheduled == numberReady`
- [ ] On smoke check timeout, the task preserves all resources and prints pod list, events, and `kubectl describe` output for both the Deployment and DaemonSet before exiting non-zero
- [ ] `mise run vind-stz-down` deletes in reverse order: DaemonSet → Deployment → RBAC → Namespace → CRD; is a no-op for absent resources
- [ ] Kubernetes manifests are in five split files under `hold-packet/deploy/vind/`: `namespace.yaml`, `crd-scaletozero.yaml` (placeholder only — replaced at deploy time by generator output), `rbac.yaml`, `hold-operator-deployment.yaml`, `hold-packet-daemonset.yaml`
- [ ] hold-packet DaemonSet runs as UID 0, `hostNetwork: true`, adds capabilities `BPF`, `PERFMON`, `NET_ADMIN`, and sets `privileged: false` / `allowPrivilegeEscalation: false`; `imagePullPolicy: IfNotPresent`
- [ ] hold-operator RBAC grants only: list/watch on `scaletozero.scale.sleepy.com` (cluster-scoped), patch/update on `scaletozero/status`; nothing else
- [ ] hold-packet DaemonSet passes `--iface $(HOLD_PACKET_IFACE)` sourced from a `HOLD_PACKET_IFACE` env var in the DaemonSet spec; default value `eth0`

**Out of scope:**
- Any behavior-specific ScaleToZero test cases — the slice ends when the environment is proved ready, not when scale-to-zero behavior is exercised
- Operator integration with gRPC or Deployment mutation — the Reconciler runs its current starter loop only
