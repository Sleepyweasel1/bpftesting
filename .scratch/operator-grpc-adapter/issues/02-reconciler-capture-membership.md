# Reconcile ScaleToZero to Captured IP membership

## What to build

Implement the Reconciler control logic in `hold-operator`. When a `ScaleToZero` event fires, the Reconciler must: (1) look up the target Deployment's associated Service ClusterIP(s) from the Kubernetes API, (2) check the Deployment's current replica count, (3) call `CaptureControl::add_rule` for each ClusterIP when replicas == 0, or `CaptureControl::remove_rule` when replicas > 0. Update `ScaleToZeroStatus` with current replica count and timestamp.

End-to-end: a `ScaleToZero` CR targeting a Deployment at 0 replicas causes the Reconciler to call `AddRule(clusterIP)` on the hold-packet daemon; scaling the Deployment to 1 replica causes `RemoveRule(clusterIP)` — observable by querying `ListRules` on the daemon.

## Acceptance criteria

- [ ] Reconciler resolves the Deployment's Service ClusterIPs from the Kubernetes API (Service selector matching).
- [ ] `add_rule` is called for each ClusterIP when replica count transitions to 0; `remove_rule` is called when count transitions to > 0.
- [ ] `ScaleToZeroStatus.replicas` and `last_updated` are written after each reconcile.
- [ ] A unit test using `FakeCaptureControl` (from the adapter issue) and a mock Kubernetes API verifies both the scale-down and scale-up paths.
- [ ] Reconciler handles Service-not-found and gRPC errors gracefully (requeue with backoff).

## Agent Brief

**Category:** enhancement
**Summary:** Teach the `ScaleToZero` reconciler to resolve all Service ClusterIPs associated with the target Deployment and keep hold-packet capture membership aligned with replica state.

**Current behavior:**
The operator already watches `ScaleToZero` resources, reads the target `Deployment`, computes `HoldCapture` vs `RemoveCapture`, updates the current `ScaleToZeroStatus` model, and calls the gRPC control plane through the `CaptureControl` trait. However, Service resolution is still a single-IP shortcut: it looks up one `Service` by the Deployment name and uses only that one `cluster_ip` string. That does not satisfy the per-destination-IP contract for selector-based Service discovery or dual-stack / multi-Service targets.

**Desired behavior:**
The reconciler should discover the `Service` objects associated with the target `Deployment` by matching each Service selector against the Deployment's pod template labels. It should collect every routable ClusterIP exposed by those Services and reconcile capture membership across that full set.

When the target Deployment has `replicas == 0`, the reconciler should ensure each resolved ClusterIP is registered with the hold-packet control plane via `add_rule`. When `replicas > 0`, it should ensure each resolved ClusterIP is removed via `remove_rule`. Missing matching Services, Services without routable ClusterIPs, Kubernetes API read failures, and transient gRPC failures should remain recoverable reconcile failures that requeue with backoff.

The status update for this slice must use the current `ScaleToZeroStatus` model already present in the codebase. Do not reintroduce legacy `replicas`, `checksum`, or `last_updated` fields. Instead, keep `intent`, `outcome`, `Reconciled` condition, `message`, and `last_reconciled_at` consistent with the new multi-IP reconciliation result.

**Key interfaces:**
- `ScaleToZeroSpec.target_ref` — continues to identify the target Deployment by name and namespace; no CRD shape change is needed
- `CaptureControl` trait — existing seam the reconciler should call once per resolved ClusterIP
- `ScaleToZeroStatus` — existing status contract; this issue should work within the current `observed_generation` / `intent` / `outcome` / `conditions` model
- The reconciler's Service-resolution logic — replace exact-name lookup with selector-based association and support multiple ClusterIPs

**Acceptance criteria:**
- [ ] The reconciler discovers Services associated with the target Deployment from Kubernetes API data by evaluating Service selectors against the Deployment pod-template labels rather than assuming Service name == Deployment name.
- [ ] All routable ClusterIPs for the matched Services are reconciled, including dual-stack or otherwise multi-IP targets.
- [ ] When replica count is `0`, `CaptureControl::add_rule` is invoked for each resolved ClusterIP; when replica count is greater than `0`, `CaptureControl::remove_rule` is invoked for each resolved ClusterIP.
- [ ] Missing matching Services, Services with no routable ClusterIP, Kubernetes API read errors, and gRPC failures produce recoverable reconcile failures and requeue.
- [ ] Reconcile status is written through the current `ScaleToZeroStatus` model and does not reintroduce the legacy `replicas` / `checksum` / `last_updated` fields described in older docs.
- [ ] Automated tests cover scale-down and scale-up behavior with multiple resolved ClusterIPs using a fake `CaptureControl` plus a mock or test seam for Kubernetes reads.
- [ ] `cargo test -p hold-operator` passes.

**Out of scope:**
