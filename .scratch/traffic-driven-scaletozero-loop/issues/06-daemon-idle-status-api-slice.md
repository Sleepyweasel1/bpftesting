## Agent Brief

**Category:** enhancement

**Summary:** Extend the daemon gRPC control plane to expose per-IP idle status evidence for captured IPs so the operator can make deterministic idle decisions.

**Current behavior:**
The daemon tracks per-IP capture state and idle evidence in map-backed state entries (`last_seen_ns`, `packet_count`, and `mode` where `Hold` indicates idle beyond timeout). The Idle Monitor computes transitions using monotonic time and idle-timeout configuration, but the gRPC API currently returns only rule membership and replay/staged-packet surfaces. The operator therefore cannot read daemon-authoritative idle evidence directly.

**Desired behavior:**
The gRPC control plane should return per-IP idle evidence for caller-specified target service IPs (IPv4 and IPv6), including enough fields for deterministic operator decisions within one polling window. Response semantics should distinguish:
- Active IP: captured and currently below/equal idle threshold.
- Idle IP: captured and above idle threshold.
- Unknown/not-captured IP: requested but absent from capture state.

Repeated reads within one poll interval should be deterministic for unchanged underlying state.

**Key interfaces:**
- `CapturelistService` protobuf contract: add RPC/messages for querying idle status evidence.
- Daemon gRPC server implementation: map state snapshots into new response types with explicit active/idle/unknown status.
- Capture-state representation (`StateEntry`): continue using authoritative fields (`last_seen_ns`, `packet_count`, `mode`) as source of truth.
- Operator control client seam: add client method(s) and type-safe parsing for new idle-status RPC responses.

**Acceptance criteria:**
- [ ] gRPC service exposes per-IP idle status and evidence fields sufficient for operator decisions (for example `last_seen_ns` or derived idle duration, threshold comparison result, and current mode/status).
- [ ] Request/response model supports target service IP sets across IPv4 and IPv6 in one API surface.
- [ ] Response includes explicit unknown/not-captured status for requested IPs that are not present in capture state.
- [ ] Repeated reads within one daemon polling interval are deterministic for unchanged state.
- [ ] Unit tests cover active, idle, and unknown/not-captured cases.
- [ ] Integration test covers end-to-end daemon RPC behavior for mixed IPv4/IPv6 target IP sets.

**Out of scope:**
- Changing idle detection algorithm, polling cadence, or timeout policy behavior.
- Implementing operator reconcile decisions that consume the new API (handled by follow-on operator slice).
- Adding new observability systems beyond the gRPC payload needed for operator decisions.
