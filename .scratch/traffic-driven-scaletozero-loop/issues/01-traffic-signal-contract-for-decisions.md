# Traffic Signal Contract for ScaleToZero Decisions

## What to build

Define and implement the operator-facing traffic signal contract used to make ScaleToZero decisions from observed traffic activity, not only replica count. The slice must produce end-to-end observable decision inputs for each target deployment and expose those inputs through ScaleToZero status messages and reconcile logs.

End-to-end: hold-packet emits traffic activity signal -> hold-operator consumes it for the target service IP set -> reconcile records decision evidence in status/message fields.

## Acceptance criteria

- [x] A stable operator-side traffic signal is available per target service IP set (active vs idle evidence) using existing daemon control-plane surfaces.
- [x] Reconcile path reads this signal and can explain decision input in status message text without changing decision semantics yet.
- [x] Unit tests cover signal present/absent and stale-signal handling as recoverable failures.

## Agent Brief

**Category:** enhancement
**Summary:** Define a stable traffic-activity signal contract so reconcile can record traffic-based decision evidence for each target service IP set without changing current decision semantics.

**Current behavior:**
Reconcile intent is currently derived from deployment replica state and records generic success/failure messaging. The daemon already exposes control-plane surfaces that can report traffic-adjacent signals (captured IP membership and staged traffic events), but reconcile does not yet normalize these into an operator-facing active/idle evidence contract.

**Desired behavior:**
Reconcile should consume a stable traffic signal per target service IP set and surface decision evidence in status/reconcile messages. This slice should be observability-first: it must not alter hold/remove/replay decision semantics yet. Missing or stale traffic signal data must be treated as recoverable failure with clear operator-visible messaging.

**Key interfaces:**
- `CapturelistService` (`ListRules`, `WatchStagedPackets`) — source surfaces for traffic evidence used by the operator-facing signal contract.
- `CaptureControl` and `StagedEvent` — operator seam for consuming daemon control-plane signal inputs.
- `ScaleToZeroStatus.message`, `ReconcileIntent`, `ReconcileOutcome`, `Reconciled` condition message — fields where decision evidence and recoverable-signal failure reasons must be made explicit.
- Reconcile assessment path — extend assessment inputs to include traffic signal presence/absence/staleness while preserving current intent semantics.

**Acceptance criteria:**
- [ ] A stable operator-side traffic signal is produced per target service IP set with explicit active vs idle evidence derived from existing daemon control-plane surfaces.
- [ ] Reconcile reads that signal and records decision input evidence in status/condition message text while keeping intent/outcome semantics unchanged.
- [ ] Signal absence and stale-signal conditions are handled as recoverable failures with deterministic messages.
- [ ] Unit tests cover signal present, signal absent, and stale-signal cases.

**Out of scope:**
- Changing scale decision semantics from replica-driven to traffic-driven in this slice.
- Introducing new daemon transport protocols or replacing existing gRPC control-plane boundaries.
- Broad policy redesign for idle thresholds or autoscaling behavior beyond decision-evidence observability.
