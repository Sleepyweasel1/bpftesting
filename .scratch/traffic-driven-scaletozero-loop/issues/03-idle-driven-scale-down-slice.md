# Idle-Driven Scale-Down Vertical Slice

## What to build

Implement idle-driven scale-down for ScaleToZero targets: when traffic remains idle past the configured threshold, reconcile scales the target Deployment to zero while keeping capture membership aligned so wake traffic can still be staged.

End-to-end: target has no traffic beyond idle window -> operator decides idle -> operator scales deployment to 0 -> capture rules remain armed for target service IPs -> status shows idle-driven decision.

## Acceptance criteria

- [x] Idle threshold policy is configurable on the ScaleToZero resource or operator config with deterministic defaults.
- [ ] When idle threshold is exceeded, reconcile scales target deployment to 0 and reports `Succeeded` with clear decision message.
- [ ] Integration/unit tests verify no false scale-down during active traffic and successful scale-down after sustained idle period.

## Agent Brief

**Category:** enhancement

**Summary:** Extend reconcile decision logic to recognize idle state from traffic signals and scale target Deployment to 0 while maintaining capture rules for wake-traffic replay.

**Current behavior:**
The reconcile loop currently makes scale decisions based on Deployment replica state and derives intent from replica count (HoldCapture when replicas > 0, RemoveCapture when replicas = 0). The traffic signal is available (from issue #01) and can report active/idle evidence per target service IP set, but reconcile does not yet use that signal to *make decisions*. The daemon's Idle Monitor already runs every 30 seconds and flips Capture Mode in the eBPF maps when `last_seen_ns` exceeds the idle timeout, but there is no operator-side policy to scale the Deployment itself once idle is detected.

**Desired behavior:**
When reconcile observes that a target service IP set has been idle for longer than the configured idle threshold (default 300 seconds matching daemon idle-timeout-secs), it should:
1. Compute a Scale intent (new or extension of `ReconcileIntent`) to scale the target Deployment to 0 replicas.
2. Execute the scale via the existing `KubeStateReader.patch_deployment_scale()` seam with safe retry behavior.
3. Record that decision in `ScaleToZeroStatus` with outcome and a message explaining: e.g., "Scaled to 0: no traffic for 300s; capture rules remain active for service IP {10.0.0.1, 10.0.0.2}."
4. If the target is already at 0 replicas and idle, report `NoOp`.
5. If already at non-zero replicas but NOT idle (active traffic detected), preserve current replicas and do not scale.

The idle detection must be deterministic and configurable. The default idle threshold in the daemon is 300 seconds; the operator must accept the same threshold (or make it configurable via ScaleToZero spec or operator environment).

**Key interfaces:**
- `ReconcileIntent` enum — consider whether to add a `Scale(i32)` variant or keep scale actions implicit in existing intents (e.g., emit Scale intent when transitioning from active to idle).
- `assess_reconcile_intent()` function — the decision point that reads deployment state and traffic signal; must now incorporate idle threshold and emit Scale intent.
- `execute_agent_intent_for_ips()` function — the execution point that applies intent to the daemon; may need extension or sibling function to handle scale before/after capture mutations.
- `KubeStateReader.patch_deployment_scale()` — already implemented; reconcile must call it with retry backoff using existing `AGENT_RETRY_BACKOFFS` pattern.
- `ScaleToZeroStatus` message field — must record the reason for scale decision, including the computed idle duration and service IPs that are idle.
- Idle threshold configuration — accept via environment variable (e.g., `IDLE_THRESHOLD_SECS`, default 300) or make it configurable on the ScaleToZero spec.

**Codebase summary:**
The reconcile loop is in `main.rs` and the decision logic is in `stz.rs`. The daemon's idle monitor is in `hold-packet/src/main.rs` and uses the same 30-second polling interval with configurable idle timeout. ADR-0004 explains the polling approach. The traffic signal seam is `CaptureControl` trait, which provides observation of current capture rules and staged packets.

**Acceptance criteria:**
- [ ] `assess_reconcile_intent()` or an equivalent decision function reads the traffic signal and computes `ReconcileIntent::Scale(0)` when idle is detected.
- [ ] Reconcile executes scale-to-0 via `patch_deployment_scale()` with retry backoff matching `AGENT_RETRY_BACKOFFS`.
- [ ] Scale operations are idempotent: scaling from N→0 when already 0, or 0→1 when already 1, reports `NoOp`.
- [ ] `ScaleToZeroStatus.message` includes the decision reason: e.g., idle duration, service IPs, threshold used.
- [ ] Capture rules remain active (`HoldCapture` intent is maintained or re-armed) while scaled to 0.
- [ ] `ReconcileOutcome` reflects `NoOp`, `Succeeded`, or `RecoverableFailure` appropriately.
- [ ] Unit tests verify: (a) no scale-down when traffic is active; (b) scale-down after idle threshold exceeded; (c) idempotent retries on transient errors; (d) status messages are accurate and observable.
- [ ] Integration test (using existing test fixtures) confirms end-to-end: active traffic → idle timeout → scale to 0 → capture rules active → wake traffic staged and replayed.

**Out of scope:**
- Changing the daemon's idle monitor polling interval or idle timeout beyond making it configurable in the operator.
- Implementing autoscaling or dynamic replica adjustment; scale decisions are binary (0 or 1 replica baseline).
- Introducing additional operator config beyond idle threshold; all other parameters remain environment variables or defaults.
- Wake-traffic orchestration or replay decision logic; that remains a separate control-plane surface.
- Structured observability (metrics, tracing) beyond status message text; status-only observability is sufficient for this slice.
