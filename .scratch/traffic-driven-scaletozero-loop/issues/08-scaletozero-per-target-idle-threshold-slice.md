## Agent Brief

**Category:** enhancement

**Summary:** Add an optional per-ScaleToZero idle-threshold configuration so each target can use a service-specific inactivity window while preserving deterministic defaults.

**Current behavior:**
Idle-threshold behavior is currently operator-global via environment configuration. `ScaleToZero` custom resources do not expose an idle-threshold field, so all targets share one threshold value. Reconcile/status messaging does not consistently distinguish threshold source (default vs per-resource override).

**Desired behavior:**
`ScaleToZero` should support an optional idle-threshold field in spec. Reconcile computes an effective threshold per target using deterministic precedence:
1. Per-resource override when present and valid.
2. Existing operator default when override is absent.

Decision logic and status messaging should include both threshold value and source (override or default) so operators can audit why scale decisions occurred.

**Key interfaces:**
- `ScaleToZeroSpec`: add optional idle-threshold field with schema/validation that preserves backward compatibility for existing resources.
- Reconcile assessment inputs: resolve effective threshold from spec + default, then pass through idle-decision path.
- `ScaleToZeroStatus.message`: include threshold source/value in decision evidence.
- CRD generation and serde schema: ensure field appears in generated CRD and is optional.

**Acceptance criteria:**
- [ ] `ScaleToZero` API/spec supports an optional per-target idle-threshold field with validation and backward-compatible defaults.
- [ ] Reconcile resolves effective threshold deterministically (spec override or operator default) per target.
- [ ] Idle decision path uses effective threshold for scale decisions.
- [ ] Status message records threshold source and value for both defaulted and overridden cases.
- [ ] Invalid threshold values are handled as recoverable failure with actionable message text.
- [ ] Tests cover default behavior, per-target override behavior, and invalid-threshold handling.

**Out of scope:**
- Reworking daemon idle-detection algorithm or polling interval.
- Changing non-threshold ScaleToZero spec fields or introducing broader policy knobs.
- Implementing full degraded-read failure semantics beyond threshold validation flow (tracked separately).
