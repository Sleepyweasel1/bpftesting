## Agent Brief

**Category:** enhancement

**Summary:** Implement operator reconcile decisions that scale workloads to zero only when daemon-authoritative per-IP idle evidence proves sustained inactivity across required target service IPs.

**Current behavior:**
The operator reconcile path resolves target deployment and service IPs, applies capture intents, and records status outcomes. It includes scale patch plumbing and retry behavior, but traffic-idle decisioning is not yet wired to daemon-authoritative idle status. The decision function currently carries an idle-threshold parameter that is not used in determining scale-down. Daemon gRPC also lacks the required per-IP idle evidence API until issue #06 lands.

**Desired behavior:**
Once daemon idle-status RPC support exists, reconcile should fetch idle evidence for target service IPs and decide scale-down only when all required IPs report idle beyond threshold. While any required IP remains active or unknown, reconcile must not scale down. When idle criteria are met, operator scales target deployment to zero idempotently and records deterministic decision evidence (threshold and relevant idle context) in ScaleToZero status while keeping capture membership armed for wake traffic.

**Key interfaces:**
- `CaptureControl` trait and concrete gRPC client: add method(s) to query daemon per-IP idle status evidence.
- Reconcile decision seam (`assess_reconcile_with_readers` and related helpers): consume idle-status responses and compute scale/no-scale intent deterministically.
- `KubeStateReader.patch_deployment_scale()`: execute zero-replica scale action with existing retry semantics and idempotent handling.
- `ScaleToZeroStatus` message and outcome fields: include clear decision evidence and preserve `NoOp`/`Succeeded`/`RecoverableFailure` semantics.
- Service-IP resolution path: ensure all required target service IPs are considered in aggregate decision logic.

**Acceptance criteria:**
- [ ] Reconcile queries daemon idle-status evidence for the full target service-IP set and blocks scale-down if any required IP is active or unknown.
- [ ] Reconcile scales deployment to 0 only when all required IPs are idle beyond configured threshold.
- [ ] Scale-to-0 execution is idempotent and reports `NoOp`, `Succeeded`, or `RecoverableFailure` consistently in status.
- [ ] Status message includes threshold and idle decision evidence sufficient for operators to understand why scale-down occurred or was blocked.
- [ ] Capture rules remain armed after scale-to-zero so wake traffic can still be staged/replayed.
- [ ] Tests cover: active traffic blocks scale-down, sustained idle triggers scale-down, and capture membership remains present while scaled to zero.

**Out of scope:**
- Implementing daemon idle-status RPC schema/server behavior (issue #06).
- Changing idle-monitor polling cadence or daemon idle-transition algorithm.
- Broader autoscaling policies beyond deterministic scale-to-zero decisions for this slice.
