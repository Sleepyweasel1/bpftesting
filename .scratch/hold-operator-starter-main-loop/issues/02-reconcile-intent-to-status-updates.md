## Agent Brief

**Category:** enhancement
**Summary:** Extend ScaleToZero reconciliation to persist intent-driven status updates after each reconcile run.

**Current behavior:**
The operator controller watches ScaleToZero resources and executes a starter reconcile loop that logs and returns `Action::await_change()`. No status writes are performed, so resource status cannot distinguish successful processing, no-op handling, or recoverable failures.

**Desired behavior:**
Each reconcile run should compute the current intent for the ScaleToZero resource and write status fields that make controller progress externally observable. Successful and no-op paths should write stable status values. Recoverable failures should surface failure intent in status while still allowing requeue/error-policy behavior.

**Key interfaces:**
- `ScaleToZeroStatus` (CRD status type) — extend/adjust fields so status can represent no-op, success, and recoverable failure outcomes.
- Reconcile function for `ScaleToZero` resources — perform idempotent status patch/update after intent evaluation.
- Kubernetes status update path (`status` subresource for the ScaleToZero CRD) — ensure writes are resilient and do not cause reconcile churn when status is unchanged.

**Acceptance criteria:**
- [ ] Reconcile updates ScaleToZero status fields on successful processing.
- [ ] Status updates include enough information to distinguish no-op, success, and recoverable failure paths.
- [ ] Repeated reconciles are idempotent and do not produce unnecessary status churn.
- [ ] Reconcile logs include the status intent/outcome written for the resource.

**Out of scope:**
- Implementing hold-packet gRPC calls (`AddRule`, `RemoveRule`, `ReplayRule`) for control-plane effects.
- End-to-end Kubernetes integration tests beyond status-update behavior for this slice.
