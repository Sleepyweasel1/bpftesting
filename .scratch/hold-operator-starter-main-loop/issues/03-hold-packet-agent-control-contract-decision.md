## Control Contract Proposal

### API surface

- Transport: gRPC (`CapturelistService`)
- Methods in scope for this slice:
	- `AddRule(ip)` for `HoldCapture`
	- `RemoveRule(ip)` for `RemoveCapture`
	- `ReplayRule(id)` for `Replay`
- Response contract:
	- Success is gRPC `OK` and `RuleResponse.success=true`
	- Any non-`OK` gRPC status or `RuleResponse.success=false` is treated as a failed mutation

### Intent to method mapping

- `HoldCapture` -> call `AddRule(target_ip)`
- `RemoveCapture` -> call `RemoveRule(target_ip)`
- `Replay` -> call `ReplayRule(staged_packet_id)`

If target IP or replay ID is not derivable from reconcile context, treat as permanent reconciliation input failure (no agent call attempted).

### Idempotency expectations

- Operator side:
	- Reconcile may retry the same intent multiple times; duplicate calls must be safe.
	- A successful retry after an uncertain prior outcome is valid and must converge status.
- Agent/daemon side behavioral requirement:
	- `AddRule` for an already captured IP is success (idempotent no-op).
	- `RemoveRule` for a missing IP is success (idempotent no-op).
	- `ReplayRule` for an already replayed/evicted packet may return an error; operator treats this as transient unless the error is a clear client input validation failure.

### Timeout, retry, and backoff

- Per RPC timeout: 2 seconds.
- Retry policy in a single reconcile execution:
	- One initial attempt + up to 2 retries for transient failures/timeouts.
	- Backoff: 200 ms then 1 s.
- After in-reconcile retries are exhausted, reconcile returns recoverable failure and requeues with controller backoff (`Action::requeue(10s)` in current loop).

### Failure classification

- Success:
	- Criteria: RPC returns `OK` and semantic success.
	- Status outcome: `Succeeded` (or `NoOp` when already reflected and no mutation needed).
- Timeout:
	- Criteria: deadline exceeded / client timeout.
	- Class: transient failure.
	- Status outcome: `RecoverableFailure`, reason/message indicate timeout.
- Transient failure:
	- Criteria: unavailable transport, connection reset/refused, temporary server errors, unknown network interruption.
	- Status outcome: `RecoverableFailure` and requeue.
- Permanent failure:
	- Criteria: invalid reconcile input (missing/invalid target identity), schema/validation mismatch proving call cannot succeed without spec change.
	- Status outcome: `RecoverableFailure` in current enum set, but message must clearly mark non-retriable input problem.
	- Reconcile behavior: do not issue repeated immediate RPC retries for the same invalid input inside one reconcile run.

### Reconcile ambiguity guardrails

- Exactly one intent is chosen per reconcile and exactly one corresponding RPC is attempted.
- Status patch is the source of truth for what was attempted and what happened (`intent`, `outcome`, `message`, `Reconciled` condition).
- When status semantics are unchanged, do not patch status to avoid churn.
- Any retry path must preserve deterministic final status for the run: either `Succeeded` or `RecoverableFailure`.
