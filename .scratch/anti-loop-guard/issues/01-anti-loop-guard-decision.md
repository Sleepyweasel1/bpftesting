# Decide and enforce Anti-loop Guard semantics

## What to build

The eBPF classifier currently contains an `is_replayed(skb.mark == 0xCAFE)` check, while replay writes raw frame bytes through TAP without setting packet mark metadata. Under the accepted Option A decision, this mark guard is treated as dead logic because recapture prevention is structural in the current topology (TC on `eth0` ingress, replay via TAP RX).

Implement Option A: remove mark-based guard behavior, document the topology invariant and correctness boundary in ADR/doc updates, and verify with integration tests.

## Decision

- Chosen option: Option A (remove the Anti-loop Guard mark check).
- Decision basis: replay via TAP re-injects frames into kernel RX and delivery continues; there is no recapture loop with TC attached only on `eth0` ingress.
- Option A invariant boundary: correctness is defined as end-to-end delivery semantics, not per-interface policy equivalence.
- Explicit non-goal: preserving original ingress-interface identity for interface-bound firewall/routing policy behavior.
- Terminology decision: keep the term Anti-loop Guard for continuity, but define it as a topology invariant under Option A.
- Canonical semantics source: ADR-0007.
- Follow-up constraint: if hook scope later expands beyond `eth0` ingress (for example, attaching on TAP ingress), re-evaluate and potentially restore explicit replay marking.
- Test strategy: keep one focused classifier behavior test for Option A invariants and retain replay end-to-end coverage as a regression guard.
